# GridTrack.Web

React-based operations dashboard for the GridTrack dispatch platform. Displays live driver positions on a map, tracks deliveries in real-time, surfaces ML-detected anomalies, and provides an analytics view over historical data. Communicates with `GridTrack.Api` over REST and SignalR WebSocket.

---


The map will work with mock data immediately. Set `VITE_USE_MOCK_SIGNALR=false` and point `VITE_API_BASE_URL` at the running backend when you're ready to connect the real pipeline.

---

## Why TypeScript, Not JavaScript

JavaScript is dynamically typed — a variable can be a string, then a number, then null, and the language won't tell you until it crashes at runtime. TypeScript adds a type layer on top: you declare what shape your data has, and the compiler enforces it before the code ever runs.

For this project specifically:
- The SignalR hub sends `DriverPositionUpdated` payloads. TypeScript forces you to define that payload's shape in `types/hub.ts`. If the backend renames a field, every component that uses it shows a red underline immediately — no runtime surprises.
- Zustand stores are typed. You can't accidentally call `liveStore.updateDriverPoistion` (typo) — TypeScript catches it before you even save the file.
- React component props are typed. If `DriverPanel` expects a `driverId: string` and you pass a number, the compiler refuses.

The cost is a bit more upfront declaration. The benefit is that bugs that would take 30 minutes to track down at runtime are caught in 5 seconds at compile time. For a project where you're the only developer and have a deadline, this tradeoff is overwhelmingly positive.

### `public/`

Static files served as-is by the dev server and bundled directly to the output root. No processing, no imports — just files.

`h3-damascus.geojson` lives here: it is a GeoJSON FeatureCollection describing ~150 H3 hexagonal cells covering the Damascus metro area at resolution 8. Generated once by `scripts/generate-h3-damascus.mjs`. The app fetches it on startup via `fetch('/h3-damascus.geojson')` and stores the result in `mapStore`.

### `scripts/`

One-off Node.js scripts that run outside the app itself. Not imported by any component.

`generate-h3-damascus.mjs` — uses `h3-js` to tile a lat/lng bounding box with H3 cells and writes the GeoJSON to `public/`. Run once. Only re-run if you change the resolution or bounding box.

### `src/app/`

Application bootstrap — the files that start everything.

| File | What it does |
|---|---|
| `main.tsx` | Entry point. Imports Leaflet CSS and global styles, mounts `<App />` into the DOM. |
| `App.tsx` | Wraps `<RouterProvider>` inside `<Providers>`. Thin wrapper. |
| `providers.tsx` | Configures every global context provider in one place: Clerk auth, React Query client, and the Sonner `<Toaster>`. If you need a new global provider, add it here. |
| `router.tsx` | Defines the URL structure using React Router. Maps each path to a page component. The `/` route wraps everything in `AppShell` so the sidebar appears on every page. |

### `src/components/`

Reusable UI pieces that are not tied to any single feature. Split into sub-directories by concern.

#### `components/ui/`

Auto-generated by the shadcn/ui CLI. **Never edit these files manually.** When you run `npx shadcn@latest add button`, it writes `button.tsx` here. The files are yours to own but shadcn won't overwrite them — re-running the CLI on the same component gives you a fresh copy you can diff against.

Contains: `button`, `badge`, `card`, `skeleton`, `table`, `sheet`, `separator`, `chart`, `sonner`.

#### `components/layout/`

The persistent chrome that wraps every page.

`AppShell.tsx` — the root layout. A flex row: sidebar on the left, `<Outlet />` (the current page) on the right filling the remaining space.

`Sidebar.tsx` — navigation links built with React Router's `<NavLink>`. NavLink automatically applies an active class when its route is the current URL, so the correct item highlights without any manual state. The Alerts link shows a badge with the count of queued anomalies pulled from `liveStore`.

#### `components/map/`

Everything that renders inside or around the Leaflet map.

`LiveMap.tsx` — the `<MapContainer>` root. Contains two internal components that must live inside MapContainer to use the `useMap()` hook: `MapRefCapture` (exposes the Leaflet map instance to the parent via a callback so it can be stored in a ref), and `AutoFollowController` (subscribes to focus mode state and calls `map.panTo` when a tracked driver moves and auto-follow is on; also detects manual panning and disables auto-follow).

`DriverMarker.tsx` — renders one Leaflet marker per driver using `L.divIcon` (an HTML div, not a PNG image). Reads all drivers from `liveStore`. When focus mode is active, the focused driver's marker is scaled up and colored amber; all others drop to 20% opacity. Clicking a marker writes the driver's ID to `mapStore` and sets the side panel mode to `'driver'`.

`HexGridLayer.tsx` — renders the H3 cell boundaries as a GeoJSON layer. Only visible when `mapStore.hexGridEnabled` is true. Clicking a cell resolves which district it belongs to, writes that district ID to `mapStore`, and sets the side panel mode to `'district'`.

`HeatmapLayer.tsx` — renders H3 cells colored by delivery density. Uses the same GeoJSON but fills each cell with a red-yellow-green color based on intensity. Implemented via the shadowed H3 cell approach (no `leaflet.heat` dependency). `interactive={false}` so clicks fall through to the hex grid layer below it.

`RoutePolyline.tsx` — a dashed amber polyline that appears only in focus mode. Reads `routePolyline` from `focusStore` and renders it as a `<Polyline>`.

`MapControls.tsx` — two icon buttons positioned absolutely over the map (top-right). Toggle hex grid and heatmap layers by writing to `mapStore`.

`ConnectionStatus.tsx` — a small indicator positioned top-left over the map. Shows a pulsing green dot when connected. Will show an amber "Reconnecting" badge when the real SignalR connection drops. Currently hardcoded to "Live" during the stub phase.

#### `components/side-panel/`

The overlay panel that slides in from the right edge of the map.

`SidePanel.tsx` — reads `sidePanelMode` from `mapStore` and renders the appropriate sub-panel. If mode is `'idle'`, renders nothing. Acts as a mode switcher, not a container with its own state.

`DistrictPanel.tsx` — shows aggregate stats for the selected H3 district: active deliveries, completed today, anomaly rate.

`DriverPanel.tsx` — shows a selected driver's name, status, and current coordinates. Contains a "Follow Driver" button that finds the driver's active InTransit delivery, builds a route polyline from mock route data, calls `focusStore.enterFocusMode`, and sets the panel to focus mode.

`FocusModePanel.tsx` — the live tracking view. Shows Order ID, driver name, a live ETA countdown (from `useEtaCountdown`), and a Lock/Unlock Camera toggle for `focusStore.autoFollow`. Exiting clears focus store and resets panel to idle.

#### `components/charts/`

Recharts-based chart components wrapped with shadcn/ui's `ChartContainer`. Each chart is self-contained: it receives data as props and handles its own tooltip and legend configuration. Built during Phase 5.

#### `components/shared/`

Generic utility components used across multiple features.

`ErrorBoundary.tsx` — a React class component (class components are required for error boundaries; hooks cannot catch render errors). Wraps the app and shows a fallback UI if a fatal error occurs, such as an unrecoverable SignalR disconnect. Built during Phase 10.

`CursorTable.tsx` — a reusable paginated table. Accepts columns config, a data array, a `nextCursor`, and `onLoadMore` callback. Used by Deliveries, Alerts, and Drivers pages to avoid duplicating table + pagination logic.

`LiveBadge.tsx` — a small styled badge component for inline status indicators.

### `src/constants/`

Static data used during the stub phase. These files are deleted or emptied when the real backend is wired.

`mockData.ts` — 15 drivers, 10 deliveries, 4 districts, and an analytics summary object. Also exports `getDistrictForCoords`, a simple spatial lookup that maps a lat/lng to one of the 4 mock districts by bounding box approximation.

`mockRoutes.ts` — 4 arrays of `[lat, lng]` pairs representing road-approximated routes through Damascus (Mezzeh → Umayyad Square, Kafr Sousa → Al-Midan, Bab Touma → Abbasiyyin, Malki → Arnous Square). The `interpolate` function densifies each route (5 steps between each pair of waypoints) so drivers appear to move smoothly. The mock emitter advances each driver one step per second along their assigned route.

### `src/features/`

Feature-scoped logic. Each subdirectory owns one page and its associated data-fetching hooks. Nothing inside a feature directory should be imported by another feature — share through `components/shared/` instead.

`live-ops/` — the map page. `LiveOpsPage.tsx` loads the GeoJSON on mount, starts the mock emitter in dev mode, and composes the map, controls, connection status, and side panel. `useFocusMode.ts` is a hook that handles the focus mode flyTo animation on entry and the Escape key exit listener.

`analytics/` — KPI cards and charts. `useAnalyticsQueries.ts` will hold the React Query calls to `/api/analytics/summary` and `/api/analytics/trends`. During stub phase it returns mock data synchronously. The chatbot tab lives in a `chatbot/` subdirectory: `ChatbotPanel.tsx` handles the date picker and message list UI, `useChatbot.ts` manages conversation history state and calls `POST /api/analysis/chat`.

`deliveries/` — cursor-paginated delivery table with status/district/date filters. Clicking a row navigates to `/` and enters focus mode for that delivery using `navigate('/', { state: { focusDeliveryId } })`.

`alerts/` — cursor-paginated anomaly alert table with date range filter.

`drivers/` — admin-only driver management table. Role-gated at the router level.

### `src/hooks/`

Custom React hooks that encapsulate reusable stateful logic. A hook is just a function whose name starts with `use` and that can call other hooks internally.

`useEtaCountdown.ts` — takes an initial `etaSeconds: number | null` and returns a formatted time string (e.g., `"7:00"`). Runs a `setInterval` that decrements every second. Returns `"--:--"` when null and `"Arrived"` when it reaches zero. Used by `FocusModePanel`.

`useDebounce.ts` — delays propagating a value until it stops changing for N milliseconds. Used by filter inputs in tables to avoid firing a new query on every keystroke.

### `src/lib/`

Infrastructure code with no UI. Organized by concern.

#### `lib/signalr/`

`hubConnection.ts` — a factory function that constructs a `HubConnection` using `@microsoft/signalr`'s `HubConnectionBuilder`. Configures the URL, automatic reconnect policy, and attaches the Clerk token via `accessTokenFactory`. Returns a singleton — calling it twice returns the same connection.

`useSignalR.ts` — a React hook that starts the hub connection on mount and tears it down on unmount. Registers event handlers: `DriverPositionUpdated` writes to `liveStore`, `DeliveryUpdated` patches `liveStore` and conditionally updates `focusStore.etaSeconds`, `AnomalyBroadcast` pushes to `liveStore.anomalyQueue` and fires a Sonner toast. Tracks connection state for `ConnectionStatus`.

`mockEmitter.ts` — DEV-only. Runs four `setInterval` loops: position tick every 1s (advances each driver one step along their route), ETA recalculation every 2s (computes remaining geodesic distance and divides by average speed), delivery status patch every 10s, anomaly injection every 15s. Returns a cleanup function that clears all intervals, called when `LiveOpsPage` unmounts.

#### `lib/api/`

`client.ts` — an axios instance with base URL from `VITE_API_BASE_URL`. A request interceptor calls `clerk.session?.getToken()` and attaches it as `Authorization: Bearer <token>`.

Each other file exports typed functions that wrap axios calls:

| File | Endpoints |
|---|---|
| `deliveries.ts` | `getDeliveries(params)`, `getDelivery(id)` |
| `drivers.ts` | `getDrivers(params)` |
| `alerts.ts` | `getAlerts(params)` |
| `analytics.ts` | `getSummary()`, `getTrends(params)` |
| `h3.ts` | `getH3Bounds()` |
| `chatbot.ts` | `sendChatMessage(history, csvData)` |

#### `lib/utils/`

`geo.ts` — thin wrappers around `@turf/turf` for operations like distance between two coordinates and bounding box computation. Keeps Turf imports centralized.

`eta.ts` — converts raw seconds to a `"M:SS"` display string. Extracted so both `useEtaCountdown` and the mock emitter's ETA calculation use the same formatting logic.

`format.ts` — date/time formatting helpers. Converts ISO timestamp strings to `"May 17, 08:30"` style display strings used in tables.

### `src/store/`

Global state, managed by Zustand. Three stores with distinct responsibilities.

`liveStore.ts` — owned exclusively by the SignalR pipeline (or mock emitter in dev). Holds the live state of every driver and delivery as flat `Record<string, T>` maps for O(1) lookup by ID. Also holds the last 50 anomaly alerts as an append-only queue. React Query never writes here.

`mapStore.ts` — pure UI state with no async. Tracks which layers are visible, which district or driver is selected, what mode the side panel is in, and the loaded GeoJSON. A component reads from this store to know what to render; it writes to this store in response to user interactions.

`focusStore.ts` — the focus mode state machine. Knows which delivery and driver are being tracked, what the current polyline and ETA are, and whether auto-follow is on. `enterFocusMode` and `exitFocusMode` are the only write paths. The map's `AutoFollowController` reads `autoFollow` to decide whether to call `map.panTo` on each driver position update.

### `src/types/`

Pure TypeScript type definitions. No runtime code — these files compile to nothing. Imported by stores, components, and API functions to share shape definitions.

| File | Contains |
|---|---|
| `driver.ts` | `DriverState` |
| `delivery.ts` | `DeliveryState`, `DeliveryStatus` |
| `district.ts` | `DistrictStats` |
| `hub.ts` | `AnomalyAlert`, `AnomalyType`, SignalR payload shapes |
| `chatbot.ts` | `ChatMessage` |

## Packages

### Runtime Dependencies

**`react` / `react-dom`** — The UI library. React lets you write components (functions that return JSX, which looks like HTML but is JavaScript). When state changes, React figures out the minimal DOM update needed. `react-dom` is the bridge between React's virtual representation and the actual browser DOM.

**`react-router-dom`** — Client-side routing. Maps URL paths to components without full page reloads. `createBrowserRouter` defines the route tree. `<Outlet />` in `AppShell` renders whichever child route is currently active. `useNavigate` is how you change routes programmatically from code.

**`@tanstack/react-query`** — Server state management. Handles the lifecycle of data that comes from an API: fetching, caching, background refetching, loading states, and error states. You define a query with a key and an async function; React Query handles the rest. Prevents you from writing `useEffect` + `useState` for every API call. Not used for SignalR data — that goes into Zustand directly.

**`zustand`** — Client state management. A minimal store library: you define state and actions in one `create()` call, then read from the store in any component with a selector. Selectors are important — they prevent a component from re-rendering when state it doesn't use changes.

**`@clerk/clerk-react`** — Authentication. Clerk handles the full auth flow (sign-in UI, token issuance, session management, token refresh). From the app's perspective: wrap the tree in `<ClerkProvider>`, use `useUser()` to read the current user's role, use `useAuth().getToken()` to get a JWT for API calls. No need to build a login page.

**`react-leaflet` / `leaflet`** — The interactive map. Leaflet is the underlying mapping library (markers, layers, zoom, pan). React-Leaflet wraps it in React components so you can write `<MapContainer>`, `<Marker>`, `<GeoJSON>`, `<Polyline>` declaratively. The `useMap()` hook gives access to the raw Leaflet map instance for imperative operations like `flyTo` and `panTo`.

**`leaflet.heat`** — A Leaflet plugin that adds a heatmap layer. Not used in the current implementation — the H3 cell color approach was chosen instead. The package is installed but the `createElementHook` wrapper was never written. Safe to remove from `package.json`.

**`@microsoft/signalr`** — The SignalR client library. Establishes a WebSocket connection (with automatic fallback to long-polling) to the ASP.NET Core SignalR hub. Exposes `connection.on('EventName', handler)` for subscribing to server-pushed messages and `connection.invoke('MethodName', args)` for calling hub methods.

**`axios`** — HTTP client for REST calls. Chosen over `fetch` because it supports interceptors — middleware that runs before every request (for attaching the auth token) and after every response (for global error handling).

**`recharts`** — The charting library. Pulled in automatically as a peer dependency of shadcn/ui's chart component. Provides `<LineChart>`, `<BarChart>`, `<ResponsiveContainer>`, etc. The shadcn `ChartContainer` wraps it with consistent theming.

**`@turf/turf`** — Geospatial operations: distance between coordinates, bounding boxes, point-in-polygon. Used via thin wrappers in `lib/utils/geo.ts`.

**`sonner`** — Toast notifications. The overlay messages that appear briefly at the top-right corner. Used to surface anomaly alerts: when the ML backend detects a stall or route deviation, the SignalR handler fires a toast with the driver name and reason.

**`next-themes`** — Theme provider that reads from `localStorage` and system preference to apply `dark` or `light` class to the HTML root. Required by shadcn/ui's Sonner wrapper.

**`radix-ui`** — Headless, accessible UI primitives (dialog, dropdown, separator, etc.). shadcn/ui components are built on top of Radix. You interact with them via the shadcn wrappers, not Radix directly.

**`clsx` + `tailwind-merge`** — Utilities for composing Tailwind class strings. `clsx` handles conditional classes. `tailwind-merge` resolves conflicts (e.g., `p-2 p-4` → `p-4`). Combined in `lib/utils.ts` as the `cn()` helper used everywhere.

**`class-variance-authority`** — Used internally by shadcn/ui components to define variant-based class sets (e.g., a button's `variant: 'default' | 'outline' | 'ghost'` maps to different Tailwind classes).

**`lucide-react`** — Icon library. SVG icons as React components. Used throughout the sidebar, buttons, and panels.

**`@fontsource-variable/geist`** — Self-hosted version of the Geist font (variable weight). Imported in `index.css`. Avoids an external Google Fonts request.

### Dev Dependencies

**`vite`** — The build tool and development server. Starts in milliseconds because it serves source files directly via native ES modules in dev mode (no bundling). Only bundles for production. Vite also handles the `@` path alias and environment variable injection.

**`@vitejs/plugin-react`** — Vite plugin that adds React-specific support: JSX transform and Fast Refresh (hot module replacement that preserves component state while you edit).

**`tailwindcss` + `@tailwindcss/vite`** — Utility-first CSS framework. Instead of writing CSS files, you compose classes directly in JSX: `className="flex h-full bg-card border-r"`. Tailwind scans your source files and emits only the CSS classes you actually use. `@tailwindcss/vite` is the Vite integration that runs this process during dev and build.

**`h3-js`** — The H3 geospatial indexing library (Uber's hexagonal grid system). Only used in `scripts/generate-h3-damascus.mjs` to tile the Damascus bounding box and output GeoJSON. Listed as a dev dependency because it's never imported by the app itself.

**`typescript`** — The TypeScript compiler. Checks types across the codebase without producing output (Vite handles the actual transpilation). Run `npm run typecheck` to check types without starting the dev server.

**`typescript-eslint`** + **`eslint`** — Static analysis. Catches common bugs (unused variables, unreachable code, incorrect types in certain patterns) and enforces code style. Run `npm run lint`.

## How a Page Renders (End to End)

To understand how the pieces connect, here is the sequence for the Live Ops page:

1. Browser loads `index.html`, which loads `src/main.tsx`.
2. `main.tsx` imports `leaflet/dist/leaflet.css` (required), then mounts `<App />`.
3. `<App />` renders `<Providers>` wrapping `<RouterProvider>`. Providers set up Clerk, React Query, and the toast system.
4. React Router matches the URL `/` and renders `AppShell`, which renders `Sidebar` + `<Outlet>`.
5. The outlet resolves to `LiveOpsPage`.
6. `LiveOpsPage` mounts. Its `useEffect` fetches `h3-damascus.geojson` and writes it to `mapStore`. Another `useEffect` starts `mockEmitter` (in dev).
7. `<LiveMap>` renders the Leaflet `<MapContainer>`. Inside it: `DriverMarkers` reads all drivers from `liveStore` and renders a `<Marker>` per driver.
8. The mock emitter's position interval fires every second. It calls `liveStore.updateDriverPosition` for each driver. Zustand triggers a re-render of `DriverMarkers`, which repositions each marker.
9. User clicks a marker → `DriverMarker`'s `click` handler writes to `mapStore` → `SidePanel` reads `sidePanelMode` from `mapStore` and renders `DriverPanel` → `DriverPanel` reads the driver from `liveStore` and renders the detail view.
10. User clicks "Follow Driver" → `focusStore.enterFocusMode` is called → `mapStore.setSidePanelMode('focus')` → `useFocusMode` detects the new `focusedDeliveryId` and calls `mapRef.current.flyTo(...)` → the map animates to the driver's position → `FocusModePanel` renders with the ETA countdown.

## Environment Variables

All variables are prefixed `VITE_` which makes them available in source code via `import.meta.env.VITE_*`. Variables without this prefix are only visible to the Node.js build process, not to the browser bundle.

| Variable | Purpose |
|---|---|
| `VITE_CLERK_PUBLISHABLE_KEY` | Clerk public key (safe to expose in browser) |
| `VITE_API_BASE_URL` | Base URL for REST calls, e.g. `http://localhost:5000` |
| `VITE_HUB_URL` | Full URL of the SignalR hub, e.g. `http://localhost:5000/hubs/dashboard` |
| `VITE_USE_MOCK_SIGNALR` | `"true"` to run the mock emitter instead of opening a real WebSocket |
